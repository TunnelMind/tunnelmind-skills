# Verify routing integrity & sovereignty

Before you route data or inference to an endpoint, check that it really is where
and who it claims. This is **GhostRoute**, TunnelMind's routing-integrity lens:
it resolves the originating ASN owner, RPKI validity, and the cert-issuing CA's
jurisdiction, then scores that reality against any sovereignty **claim** it can
recover for the subject.

## When to run this

- You are about to send data/inference to an endpoint that asserts a
  **jurisdiction** — EU data residency, FedRAMP, sovereign-AI.
- You want to detect a **US-fronted** (Cloudflare/AWS/GCP) endpoint masquerading
  as sovereign-EU infrastructure.
- You need to catch an **RPKI-invalid origin** (possible BGP hijack) or a
  **sanctioned operator** before connecting.

## How to call

Prefer the MCP tool if `mcp-data.tunnelmind.ai` is connected:

```
ghostroute_check({
  entity: "api.mistral.ai",
  receipt: false,   // optional
})
```

Otherwise call the HTTP endpoint directly (the entity is in the path):

```
GET https://data.tunnelmind.ai/v1/ghostroute/check/api.mistral.ai
```

### Request fields

| field | required | meaning |
|-------|----------|---------|
| `entity` | yes | An IPv4/IPv6 address, domain, ASN (`AS####`), or cert SHA-1/256 to check. |
| `receipt` | no | When `true`, issues a signed, persisted GhostRoute receipt (`GR-YYYY-NNNNNNN`) instead of an ephemeral verdict. |

## How to read the result

`sovereign_tier` is the headline (or `null` when the subject makes no
sovereignty claim to check against):

| `sovereign_tier` | What it means |
|------------------|---------------|
| `VERIFIED` | Routing/ownership/jurisdiction match the claim. Safe to rely on at the stated integrity. |
| `PLAUSIBLE` | No contradiction, but the claim is not fully corroborated. Treat as unconfirmed. |
| `MISMATCH` | Reality diverges from the sovereignty claim (e.g. a "EU-sovereign" service fronted from US infrastructure). Do not rely on the claim. |
| `CRITICAL_MISMATCH` | A severe divergence or a tripped hard floor. Do not route sensitive data here; report why. |

Corroborate with `sovereign_integrity` (a [0,1] score) and the evidence:
`origin_as`, `rpki_status`, `cert_ca`, `claimed_sovereign_zone`,
`is_ai_infrastructure`, `ai_owner`, `sanctions_match`. Always read
`_meta.caveats` and `_meta.penalties` — they say what was and wasn't
determinable, so you don't over-read a thin result.

Hard floors override the headline score — if `signals`/`reasons` mention `ghostroute.rpki=INVALID` (possible BGP hijack) or `sanctions_match`, treat it as the worst verdict regardless of the number.

## Issue a durable receipt when it matters

By default the verdict is ephemeral. Pass `receipt: true` to get a signed,
persisted GhostRoute receipt (`GR-YYYY-NNNNNNN`) instead — attach its id to
your audit log as re-checkable proof of the routing reality you observed.
