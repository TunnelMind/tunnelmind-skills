# Reconcile an actor (claim vs conduct)

An actor has presented a **cryptographic key** and made a claim about itself.
This is the **reconciliation** call — it answers "does what this key claims
about itself match what the network has seen its subject do?" Distinct from
`verify-actor` (what is this network destination?) and `preflight-should-i-act`
(may I act *now*?).

## When to run this

- An agent-to-agent handshake where the counterparty signs with an Ed25519 key
  or `did:key` and asserts a tier or capability.
- You need to confirm a **claimed attestation** (a RATS/EAT hardware/platform
  attestation) is real and anchored — not self-asserted.
- You suspect a key is **claiming more than it can prove** (over-attestation).

## How to call

Prefer the MCP tool if `mcp-data.tunnelmind.ai` is connected:

```
verdict_lookup({
  key: "did:key:z6MkhaXgBZDvotDkL5257faiztiGiC2QtKLGpbnnEGta2doK",
  subject: "nytimes.com",   // optional
  nonce: "...",   // optional
  sig: "...",   // optional
  eat: "...",   // optional
  claims: "...",   // optional
})
```

Otherwise call the HTTP endpoint directly (the key is in the path; pass the
subject it claims to act as so the conduct side can be reconciled):

```
GET https://data.tunnelmind.ai/v1/verdict/did:key:z6Mk…?subject=nytimes.com
```

### Request fields

| field | required | meaning |
|-------|----------|---------|
| `key` | yes | The actor's Ed25519 public key, as hex (64 chars, optional 0x), base64url (43 chars, unpadded), or did:key (did:key:z6Mk…). |
| `subject` | no | An ip / domain / ASN / entity_slug the key claims to act as. Drives the conduct (graph behavior) side of the reconciliation. Omit for a claim-only verdict. |
| `nonce` | no | Binding nonce for a bare-Ed25519 self-attestation (paired with `sig`). |
| `sig` | no | Base64 Ed25519 signature over `nonce`, proving control of the key. |
| `eat` | no | A RATS/EAT compact JWS (EdDSA) attesting this key, signed by a trusted anchor. |
| `claims` | no | URL-encoded JSON array of raw claim objects, e.g. `[{"type":"ed25519-self","signature":"…","nonce":"…"},{"type":"eat","token":"…"}]`. Overrides the `nonce`/`sig`/`eat` convenience params when present. |

## How to read the result

- **Attestation ladder** — `claim.best_verified_tier` is the strongest tier the
  key actually *proved*: `self-asserted` < `software` < `tee-tpm` < `silicon-root`. It is always measured/anchored; a token
  claiming a higher tier than its trust anchor permits is **capped down**, never
  taken at face value. `claim.key_control_proven` tells you whether control of
  the key was actually demonstrated (via `nonce`+`sig`).
- **Contradictions are the headline** — `reconciliation.contradictions` is the
  reason to care. **Any entry means do not rely on the claim**; report which:

| contradiction | meaning |
|---------------|---------|
| `claim_exceeds_conduct` | The key attests a higher trust tier than its observed behavior supports (e.g. claims `silicon-root` but behaves as a low-trust node). |
| `unverified_claim` | A claim was presented but failed to verify against its trust anchor. |
| `key_control_unproven` | Claims were presented without proving control of the key (no valid `nonce`/`sig` binding). |

- **`reconciliation.verdict`** carries `tier`, `reputation`, `flags`, and
  `confidence`; weight your conclusion by `confidence`. `claim_vs_conduct_delta`
  quantifies the gap between what was claimed and what was observed.
- A key is linked to an identity **only when control is cryptographically
  proven** — never inferred from behavioral correlation. Omitting `subject`
  yields a claim-only verdict (no conduct side).

## Keep the self-verifying receipt

The response includes a TunnelMind **`receipt`** that commits to the
reconciliation. It is *self-verifying*: given the receipt bytes plus the inline
witness public keys, anyone re-derives the verdict and checks log inclusion
**offline** — no call back to TunnelMind. Attach it as portable evidence of what
the key claimed and what it could actually prove.
